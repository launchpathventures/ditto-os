# ADR-043: Project Substrate via `.ditto/` Directory in Target Repo

**Date:** 2026-04-27
**Status:** accepted

## Context

Ditto manages projects on behalf of users — each project has a runner kind, a
trust tier, role contracts, a skill index, a tool allowlist, project-specific
guidance, and a human-readable onboarding report. Brief 224 (Project
Onboarding & Battle-Readiness, parent brief) decomposed this into four
sub-briefs; sub-brief #3a (Brief 228) ships the **Project Retrofitter** —
the component that writes the substrate INTO the target repo at connection
time and on demand.

Three architectural questions this ADR resolves:

1. **Where does the substrate live?** In Ditto's DB? In the target repo's
   working tree? Both? Insight-201 (user-facing data seams default to
   filesystem-like legibility) says: file-as-primary at user-facing seams.
   The substrate is user-facing — the user grep-s it, edits it, diffs it,
   commits it. So it lives in the target repo's working tree.

2. **What's the directory shape?** `.catalyst/` already exists as a
   competing convention (per Brief 215's `harnessType: "catalyst"`). Ditto
   needs its own namespace that doesn't conflict. `.ditto/` is the natural
   name; sibling-by-convention to `.git/`, `.github/`, `.claude/`,
   `.catalyst/`.

3. **What happens when a user edits a Ditto-regenerated file?** Naive
   re-runs would silently overwrite — a real production-data-loss path
   (Brief 228 Reviewer CRIT-3, originally surfaced as the brief's §Open
   Question Q3). This ADR defines the user-edit safety mechanism.

## Decision

### `.ditto/` Directory Shape (v1)

Ditto-managed projects carry a `.ditto/` directory at the target repo root,
with the following structure:

```
.ditto/
├── role-contracts/
│   ├── dev-pm.md
│   ├── dev-architect.md
│   ├── dev-builder.md
│   ├── dev-reviewer.md
│   ├── dev-documenter.md
│   ├── dev-researcher.md
│   └── dev-designer.md
├── skills.json
├── tools.json
├── guidance.md
├── onboarding-report.md
├── version.txt
└── .gitignore
```

Each file's purpose:

- **`role-contracts/<role>.md`** — markdown role-contract, one per
  `/dev-*` role, derived from `.claude/commands/<role>.md` but rewritten
  in project-context (project name, build/test commands, runner kind,
  trust tier).
- **`skills.json`** — skill index: `{ version, skills: [{ name, scope, sourceCommit }] }`.
  MVP: `skills` empty array; future briefs populate.
- **`tools.json`** — tool allowlist: `{ version, allowed: string[], denied: string[] }`.
  MVP: `allowed` = built-in safe tools (Read, Grep, Glob, Edit, Write, Bash);
  `denied` = empty.
- **`guidance.md`** — project-specific guidance: build commands, test
  commands, branch naming, "things that have surprised past contributors."
- **`onboarding-report.md`** — human-readable analyser report (mirrors the
  Brief 226 `AnalyserReportBlock` content).
- **`version.txt`** — schema version (currently `1`); future ADRs ship a
  migrator.
- **`.gitignore`** — guidance for what NOT to commit (per-developer
  overrides; cache files; future brief territory).

### Header Convention

Every Ditto-regenerated markdown / JSON file starts with a header marking
it as Ditto-managed:

- Markdown files: `# DO NOT EDIT — regenerated by Ditto retrofit (run <processRunId>). See ADR-043.`
- JSON files: top-level `$generatedBy` field with the same text (JSON has
  no comment syntax).

The `<processRunId>` is the opaque UUID of the retrofit run that produced
the file — grep-able from the user's git log, cross-referenceable to
Ditto's `harness_decisions` audit row.

`version.txt` is the exception — it's a single-line file holding only
the schema version integer. The audit trail for `version.txt` lives in
the commit message + harness_decisions, not the file itself.

### Catalyst Coexistence

Ditto's `.ditto/` is for projects opting INTO Ditto-driven onboarding. It
does NOT replace existing harness substrates:

- A project may have BOTH `.catalyst/` AND `.ditto/`.
- `.ditto/` writes never touch sibling directories (`.git/`, `.github/`,
  `.claude/`, `.catalyst/`, etc.).
- The retrofitter's prompt template (Brief 228 `retrofit-prompt.ts`)
  forbids modifications outside `.ditto/`.

### Versioning

`.ditto/version.txt` carries the current schema version (`1` for Brief
228 / ADR-043). When a future ADR revises the shape:

- Bump the version integer.
- Ship a migrator script alongside the new ADR.
- The retrofitter detects the prior version + applies the migrator before
  writing new files.

Migrator tooling is OUT OF SCOPE for Brief 228 — only the version field
ships. The first `.ditto/version.txt` files all read `1`.

### Re-Run Discipline (Idempotency + User-Edit Safety)

Brief 228 ships the retrofit pipeline with two layers of safety:

1. **`unchanged` files (Insight-212 idempotency).** The plan generator
   computes a sha256 hash of each file's planned content. Files whose
   current on-disk hash matches the plan's hash are marked `'unchanged'`
   and excluded from the dispatch payload. Re-runs that produce only
   `'unchanged'` files complete with `commitSha=null` and an info
   AlertBlock side-car ("no changes to retrofit").

2. **`skippedUserTouchedFiles` (Q3 resolution).** For each `'update'`
   file (where the on-disk hash differs from the plan):
   - The runner-side agent compares the on-disk content against the
     PRIOR retrofit's hash (read from
     `harness_decisions.reviewDetails.retrofit.fileHashes`).
   - If the on-disk hash differs from BOTH the prior-retrofit hash
     AND the plan hash, the file was user-edited between retrofits.
   - **Under autonomous tier**: user-touched files are EXCLUDED from the
     write set. The runner returns them in `skippedFiles` (with the
     suffix ` (user-edited)`). The `verify-commit` handler populates
     `RetrofitPlanBlock.skippedUserTouchedFiles`; the renderer surfaces
     an info AlertBlock side-car listing the skipped files.
   - **Under spot_checked + supervised tiers**: the user explicitly
     approves the overwrite — no automatic skip.

The `# DO NOT EDIT` header is a SECONDARY safety signal beyond hash-
compare. The hash-compare is authoritative; the header is informational
(and protects against the rare hash-collision edge case where a
user-edited file accidentally hashes to the prior content).

## Provenance

- **`.ditto/` directory shape:** Original to Ditto. Brief 224
  §"Architectural Decisions Captured / `.ditto/` Directory Shape"
  outlined the candidate structure; this ADR finalises it.
- **Header convention:** Original to Ditto. Standard hidden-config-
  directory pattern (sibling-by-convention to `.git/`, `.claude/`,
  etc.).
- **User-edit safety mechanism (hash-compare + skip):** Original to
  Ditto. Reviewer-recommended in Brief 228's CRIT-3 resolution. The
  hash-compare approach is canonical for filesystem-projection systems
  (cabinet, dotfiles managers, etc. all use it); naming + integration
  with Ditto's runner/trust system is original.
- **Versioning seam:** Original to Ditto. Mirrors `package.json`
  `"version"` semantics + standard schema-versioning practice.
- **Catalyst coexistence stance:** Per Brief 224 §Constraints — Ditto
  takes a "siblings, not replacement" posture toward existing harness
  substrates.

## Consequences

### What becomes easier

- **Project legibility for the user.** The user grep-s `.ditto/`
  directly — the substrate IS in the repo, not opaque in Ditto's DB.
  Insight-201 application #2 (Brief 199's memory projection is #1).
- **Audit-trail integrity.** Every retrofit's commit message carries
  `(run <processRunId>)`; cross-referencing Ditto's
  `harness_decisions` row from the user's git log is one grep away.
- **Multi-project consistency.** Every Ditto-managed project has the
  same `.ditto/` shape — agents working on different projects find the
  same files in the same places.
- **Catalyst migration path.** Projects already using `.catalyst/` can
  ALSO opt into `.ditto/` without conflict — the two coexist.

### What becomes harder

- **`.ditto/` schema evolution requires migrators.** Once v1 ships,
  changing the directory shape requires a version bump + a migrator. We
  can't silently rename or remove files — they're in user repos, in git
  history. The migrator complexity is real (deferred to a future brief
  for now).
- **Runner-side trust budget.** Each retrofit dispatch sends
  ~10-20 KB of file content to the runner. For cloud runners
  (claude-managed-agent etc.), this counts against the runner's token
  budget. Mitigation: the prompt references the structured payload by
  KEY (`retrofitDispatch` in the workItem context), not inline content.

### What new constraints does this introduce

- **`.ditto/` writes NEVER touch sibling directories.** Hard rule;
  enforced by the runner prompt (Brief 228 `retrofit-prompt.ts`).
- **Header convention is enforced by the plan generator.** Every
  regenerated markdown/JSON file gets the header at compose time. The
  header includes the `processRunId` — generators that omit it produce
  files that fail the user-edit safety check (the header itself
  changes the hash, so a re-run would always mark the file as "update"
  unless we exclude the header from the hash, which we don't).
- **Schema version is monotonic.** `.ditto/version.txt` integers go up,
  never down. Future migrators handle downgrade rejection.

### What follow-up decisions are needed

- **`.ditto/` schema v2** (when needed): a future ADR + migrator. Likely
  drivers: skills.json schema evolution; per-developer-override directory
  shape (`.ditto/local/`); cache file conventions (`.ditto/cache/`).
- **Multi-user `.ditto/` ownership**: per-developer overrides require
  conflict-resolution policy. Out of scope for Brief 228 (solo-user
  shape per pipeline-spec).
- **Scheduled cron-driven re-run**: Brief 228 ships only the on-demand
  re-run path (button + Self tool); cron-driven re-run is a follow-on
  enhancement.
- **`.catalyst/` ↔ `.ditto/` interop**: when a project has both, can
  Ditto's `.ditto/role-contracts/` reference `.catalyst/` artefacts?
  Future ADR if interop demand justifies.
