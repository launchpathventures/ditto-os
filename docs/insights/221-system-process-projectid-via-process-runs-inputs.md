# Insight-221: System processes carry `projectId` in `processRuns.inputs`, not in `processes.projectId`

**Date:** 2026-04-28
**Trigger:** Brief 228 (Project Retrofitter) Reviewer CRIT-2 — three queries in the Brief 228 codebase (`readPriorRunHashes`, the `rerun_project_retrofit` Self tool's last-known-tier lookup, the `/api/v1/projects/[id]/retrofit` route's last-known-tier lookup) filtered on `processes.projectId` thinking that's where the project-scope filter belongs. Reality: system processes (loaded via `process-loader.ts`) have `processes.projectId = null` — they're shared across all projects. The per-run project scope lives in `processRuns.inputs` (the runtime input map, populated when `startProcessRun(slug, { projectId: ... }, ...)` is called). All three queries returned zero rows for any project; idempotency + last-known-tier features were broken in production-shaped data.
**Layers affected:** L1 Process (process loader), L3 Harness (process_runs invocation), L5 Learning (memory-projection-style queries)
**Status:** active

## The Insight

The `processes` table has TWO kinds of rows:
- **System processes** — loaded by `process-loader.ts` from `processes/*.yaml` at boot. Shared across all projects. `projectId = null`. Examples: `project-onboarding`, `project-retrofit`, `morning-briefing`.
- **User-authored processes** — created by the user via the process editor or generated by a Self tool. Bound to a specific project. `projectId = <project.id>`. Examples: per-project content workflows, custom agent rituals.

A system process's runs carry their per-run project scope in `processRuns.inputs` — populated when the trigger surface calls `startProcessRun(slug, { projectId: <id>, ... }, triggeredBy, options)`. The `projectId` lives ONLY in the JSON inputs map; querying `processes.projectId` for system-process runs always returns null.

This means: code that filters processRuns / harness_decisions / step_runs by project must use the right filter for the right kind of process:
- **User-authored processes:** filter via `processes.projectId = <id>` (joining processes through processRuns.processId).
- **System processes:** filter via `json_extract(processRuns.inputs, '$.projectId') = <id>` (the SQLite-drizzle pattern; verified at `src/engine/inbound-email.ts:526` + `src/engine/network-chat.ts:557` for similar use cases).

Mixed queries (filter on EITHER) need an `OR` predicate joining both conditions.

## Implications

1. **Future code querying processRuns / step_runs / harness_decisions by project must pick the right filter.** The choice is: am I querying user-authored processes only, system processes only, or both?

2. **The pattern Brief 228 codified:**
   ```ts
   .where(
     and(
       eq(schema.processes.slug, "project-retrofit"),  // pick the system process by slug
       sql`json_extract(${schema.processRuns.inputs}, '$.projectId') = ${projectId}`,
     ),
   )
   ```
   For any system process whose YAML declares `projectId` as a runtime input, this is the canonical filter.

3. **The trap is subtle.** `processes.projectId` SCHEMA-EXISTS — it's a real notNull column. The drizzle query type-checks fine. The bug only surfaces in production-shaped data where the filter returns zero rows; unit tests that seed user-authored processes (with `projectId` set) miss the system-process case entirely.

4. **Idempotency + tier-defaulting features depend on this.** Brief 228's idempotent re-run reads prior `harness_decisions` rows to detect `'unchanged'` files; the Self tool + retrofit route default the trust tier from the most recent prior run. Both feature paths were silently inert in production until the CRIT-2 fix.

5. **Reviewer discipline:** specifically check the project-scope filter's correctness in any query that touches a system process. The reviewer caught Brief 228's three callsites by tracing the data flow. Future reviews should take the same posture for any new query joining `processes` and `processRuns` for project-scoped retrieval.

## Where It Should Land

**Near-term:** Add a one-paragraph note in `docs/architecture.md` §Layer 1 (Process) — "System vs user-authored processes" — clarifying that system processes have `projectId = null` at the table level, and per-run scope lives in `processRuns.inputs`.

**Medium-term:** Consider a small reusable utility at `packages/core/src/db/process-run-queries.ts` exporting helpers like `processRunsByProject(projectId, processSlug?)` that build the right WHERE clause for the system-vs-user-authored case. Brief 228 could have used this if it existed; future briefs will need it.

**Long-term:** If a third or fourth code path repeats the mistake, escalate to a query-layer abstraction (drizzle relation hooks, a custom drizzle plugin, etc.) that hides the filter choice. Today's posture — every callsite picks explicitly — is verbose but visible.

**Status until absorbed:** active. Will become absorbed when (a) the architecture.md §L1 note lands AND (b) the helper utility exists in core. Trigger for absorption: third query that needs the pattern.
