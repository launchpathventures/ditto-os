## 1. Reflexion (Shinn et al., NeurIPS 2023)

### What it is

Reflexion is a framework for "verbal reinforcement learning" where language agents reflect on task feedback signals and store those reflections in an episodic memory buffer to improve performance on subsequent trials.

### What is stored

Natural language self-reflections — verbal descriptions of what went wrong and what to try differently. These are free-form text strings generated by a dedicated Self-Reflection model (which can be the same LLM as the actor).

### Storage mechanism

An episodic memory buffer that accumulates reflections across trial iterations. The buffer is a simple list/array of text strings. There is no vector database or embedding — reflections are stored as raw text and injected directly into the prompt.

### Retrieval mechanism

Retrieval is controlled by a `ReflexionStrategy` enum with four modes:

| Strategy | What is provided to the agent |
|----------|-------------------------------|
| `NONE` | No information about previous attempts |
| `LAST_ATTEMPT` | The reasoning trace from the last attempt |
| `REFLEXION` | The self-reflection text from the last attempt |
| `LAST_ATTEMPT_AND_REFLEXION` | Both the reasoning trace and the self-reflection |

Reflections are concatenated into the prompt context for subsequent trials. There is no semantic search — the most recent reflection(s) are appended directly.

### What triggers storage

A trial failure triggers reflection generation. The agent receives feedback (binary success/fail, or a scalar reward from an evaluator), and if the task is not solved, the Self-Reflection model generates a verbal reflection that is appended to the memory buffer.

### Known limitations

Reflexion is vulnerable to **degeneration-of-thought**: the agent repeats the same flawed reasoning across iterations even when explicit failures are identified. Two documented failure modes:

1. **Confirmation bias in self-reflection**: When the actor commits an initial reasoning error, the reflection often restates the incorrect reasoning and justifies the error through additional flawed logic rather than isolating the root cause.
2. **Mode collapse across iterations**: Because the same model generates, evaluates, and revises reasoning, corrections follow a narrow, repetitive template. The model cannot escape its own entrenched reasoning patterns.

The root cause: self-reflective approaches depend on fixed internal thinking patterns. Even when prompted with different personas, underlying reasoning strategies remain uniform. This is the "EMNLP 2025 caveat" — LLMs struggle to self-correct without external grounding. Multi-Agent Reflexion (MAR) addresses this by replacing single-agent self-critique with structured debate among diverse persona-based critics.

## 2. Anthropic Harness Architecture (Nov 2025 Blog)

### What it is

A two-phase agent harness for multi-session software development. An **initializer agent** sets up the project environment on the first run; a **coding agent** makes incremental progress session-by-session, reading and updating shared state files.

### What is stored

Three persistent artifacts:

| Artifact | Format | Purpose |
|----------|--------|---------|
| `claude-progress.txt` | Free-form text log | Running record of what agents have done, what's working, what's left |
| `feature_list.json` | JSON array of objects | Comprehensive, testable feature requirements with pass/fail status |
| `init.sh` | Shell script | Starts the development server for end-to-end testing |

The feature list uses JSON deliberately — "the model is less likely to inappropriately change or overwrite JSON files compared to Markdown files." Each feature entry has this structure:

```json
{
  "category": "functional",
  "description": "New chat button creates a fresh conversation",
  "steps": [
    "Navigate to main interface",
    "Click the 'New Chat' button",
    "Verify a new conversation is created"
  ],
  "passes": false
}
```

Coding agents are prompted to edit feature_list.json **only by changing the `passes` field** — never modifying descriptions or steps.

### Session loop

Each coding agent session follows a fixed pattern:

1. **Orient**: Run `pwd`, read `claude-progress.txt`, read `feature_list.json`, read `git log --oneline -20`
2. **Verify**: Start the dev server via `init.sh`, run basic end-to-end tests to catch undocumented regressions
3. **Select**: Choose the highest-priority feature that is not yet passing
4. **Implement**: Write code, test iteratively
5. **Commit**: Git commit with descriptive message, update `claude-progress.txt`, mark passing features in JSON
6. **Recover**: If code breaks, use `git revert` to recover the last working state

### What triggers storage

- Progress file: updated at end of every session
- Feature list: `passes` field flipped from `false` to `true` when a feature's test steps succeed
- Git commits: created after each meaningful code change

### Key design insight

Git history + progress file together provide redundant state. Git gives precise code diffs; the progress file gives human-readable narrative. The initializer generates 200+ features for a complex app — the feature list acts as a persistent, structured backlog that survives across unlimited sessions.

## 3. Devin (Cognition)

### What is stored

Two distinct persistence layers:

**Layer 1 — In-session notes (`notes.txt`)**: Devin logs thoughts during a session in a notes file: "User wants X," "We're doing Y," "I hit issue Z, working on it." These serve as working memory that Devin references in later steps within the same session to maintain continuity.

**Layer 2 — Cross-session Knowledge base**: A structured collection of persistent knowledge items, each comprising:
- **Trigger description** (required): A phrase describing when this knowledge should be recalled
- **Content**: A handful of sentences with relevant information
- **Macro** (optional): A short `!identifier` (e.g., `!deploy-checklist`) for explicit invocation in prompts
- **Scope**: Pinned to no repo, a specific repo, or all repos
- **Enabled/disabled toggle**: Per-user control

### Retrieval mechanism

Knowledge retrieval is trigger-based: "Devin will retrieve a Knowledge item when its current work is related to the specified triggers." The system matches current work context against trigger descriptions automatically. Macros provide an explicit invocation path.

### What triggers storage

- **User-created**: Manual entry via the Knowledge tab in Settings
- **Auto-suggested**: Devin proactively suggests Knowledge items based on user feedback in chat. Users can edit, regenerate, or dismiss suggestions.
- **Repo knowledge**: Devin auto-generates and auto-updates its own Knowledge on repo structure and components

### Scheduled session continuity

For recurring tasks (e.g., weekly release notes), Devin reads and writes its own notes across sessions so each run builds on the prior run's context. "It won't re-summarize the PRs it already covered last week because Devin knows where it left off."

### Known limitations

Cognition's own engineering team found that when they tested removing their memory infrastructure and relying solely on model-generated notes (with Sonnet 4.5), they "saw performance degradation and gaps in specific knowledge." The model's self-generated summaries "weren't comprehensive enough" and sometimes omitted critical task details. Their existing compacting and summarization systems proved more effective than model-generated state. This is a direct parallel to Reflexion's degeneration-of-thought finding.

Additionally, as of mid-2025, Devin "does not maintain long-term memory across sessions" in the traditional sense — its cross-session continuity relies on the Knowledge base and notes mechanism rather than true persistent memory.

## 4. AutoGPT / AgentGPT

### Memory backends (historical)

AutoGPT supported multiple memory backends:

| Backend | Mechanism | Status |
|---------|-----------|--------|
| **LocalCache** (default) | JSON file (`auto-gpt.json`) | Still available |
| **Redis** | In-memory store (default with Docker Compose) | Still available |
| **Pinecone** | Vector database | Removed in vector memory revamp |
| **Weaviate** | Vector database | Removed |
| **Milvus** | Vector database | Removed |

### The vector database removal

AutoGPT underwent a "vector memory revamp" that removed all vector database implementations, leaving only a couple of classes responsible for memory management with JSON files as the default. The rationale (per Jina.AI founder Han Xiao): vector databases are "an overkill solution" because LLM inference (10+ seconds) is the actual bottleneck, not vector search. Numpy's `dot` product takes under 5ms even for 100,000 embeddings — approximate nearest neighbor search adds complexity without meaningful speedup.

### Current storage mechanism

Memories and their embeddings are stored in a JSON file. Retrieval uses `numpy.dot` for cosine similarity comparison against stored embeddings. The architecture is: text -> embedding -> store in JSON -> retrieve via brute-force dot product.

### Known reliability issues

- The JSON memory file clears if AutoGPT is exited or crashes
- Data ingestion can cause the JSON file to revert to ~1KB, losing ingested data
- Every new run historically started from scratch on its task

### What triggers storage

The agent stores memories during its think-execute cycle. Each action-observation pair can generate a memory entry. The `memory_add` command stores text with its embedding; `memory_get` retrieves relevant memories by embedding similarity.

## 5. Cursor Rules / cursor-memory

### What is stored

Two mechanisms for cross-session knowledge:

**Project Rules** (`.cursor/rules/`): Markdown files (`.mdc` format) with optional YAML frontmatter for path scoping. Each rule file covers one topic with a descriptive filename. Files are version-controlled.

Example frontmatter:
```yaml
---
paths:
  - "src/api/**/*.ts"
---
```

**Legacy `.cursorrules`**: A single text file in the project root that acts as a system prompt prepended to every conversation. Now deprecated in favor of `.cursor/rules/`.

### Format

Rules use MDC (Markdown Components) format — standard markdown with metadata frontmatter. The frontmatter supports:
- `description`: What the rule is about
- `paths`: Glob patterns for conditional loading (e.g., `"src/**/*.{ts,tsx}"`)
- `alwaysApply`: Boolean to force loading regardless of context

Rules without `paths` frontmatter load unconditionally at session start. Path-scoped rules trigger when Cursor reads files matching the pattern.

### How it is loaded into context

Rule contents are "included at the start of the model context." All unconditional rules load at launch. Path-scoped rules load on demand when the agent reads matching files. This is a static injection — not semantic retrieval.

### Community "Memory Bank" pattern

The community has developed a "Memory Bank" pattern using `.cursor/rules/` files to simulate persistent project memory. This involves creating numbered markdown files (e.g., `00_description.md`, `01_architecture.md`) in a `.cursor/memory/` folder, with a rule that instructs Cursor to consult these files. This is a user-authored convention, not a built-in feature.

### What triggers storage

Rules are manually authored by developers. There is no auto-generation or learning mechanism built into Cursor itself. Context resets every session — rules are the only built-in persistence mechanism.

## 6. Claude Code Memory System

### What is stored

Two complementary systems:

| System | Author | Contents | Scope | Loaded |
|--------|--------|----------|-------|--------|
| **CLAUDE.md files** | Human | Instructions, rules, project architecture | Project, user, org, or managed policy | Every session, in full |
| **Auto memory** | Claude | Build commands, debugging insights, preferences, patterns | Per working tree (git repo) | Every session (first 200 lines / 25KB of MEMORY.md) |

### CLAUDE.md hierarchy

| Scope | Location | Shared with |
|-------|----------|-------------|
| Managed policy | `/Library/Application Support/ClaudeCode/CLAUDE.md` (macOS) | All org users |
| Project | `./CLAUDE.md` or `./.claude/CLAUDE.md` | Team via source control |
| User | `~/.claude/CLAUDE.md` | Just you, all projects |

CLAUDE.md files walk up the directory tree from the working directory. Subdirectory CLAUDE.md files load on demand when Claude reads files in those directories. Files support `@path/to/import` syntax for importing additional files (max depth: 5 hops).

### Path-scoped rules

`.claude/rules/` directory holds markdown files, each covering one topic. Files support YAML frontmatter with `paths` field for conditional loading:

```yaml
---
paths:
  - "src/api/**/*.ts"
---
```

Rules without `paths` load unconditionally at launch. Path-scoped rules trigger when Claude reads matching files.

### Auto memory structure

Stored at `~/.claude/projects/<project>/memory/`:

```
~/.claude/projects/<project>/memory/
├── MEMORY.md          # Concise index, loaded every session
├── debugging.md       # Detailed notes (loaded on demand)
├── api-conventions.md # Topic files Claude creates as needed
└── ...
```

`MEMORY.md` acts as an index. First 200 lines or 25KB are loaded at session start. Claude moves detailed notes into separate topic files to keep the index concise. Topic files are read on demand via standard file tools.

### What triggers storage

Claude decides what to save based on whether information "would be useful in a future conversation." It stores build commands, debugging insights, architecture notes, code style preferences, and workflow habits. Not every session produces a memory write.

### Retrieval mechanism

Static injection — `MEMORY.md` contents are loaded into the context window at conversation start. Topic files are read on demand. There is no embedding or semantic search. HTML comments in CLAUDE.md files are stripped before injection to save tokens.

### Effectiveness data

Files under 200 lines achieve a rule application rate above 92%, compared to 71% beyond 400 lines. CLAUDE.md content is delivered as a user message after the system prompt, not as part of the system prompt itself.

## 7. Windsurf / Codeium Cascade

### What is stored

Two mechanisms:

**Memories**: Auto-generated context objects created by Cascade during conversation when it "encounters context that it believes is useful to remember." Users can also explicitly request memory creation.

**Rules**: Manually defined behavioral instructions at three scopes — global, workspace, and system level (for enterprise teams). Rules are version-controlled and shareable. The recommended format is to use an `AGENTS.md` file in the repo root.

### Storage

Memories are stored locally at `~/.codeium/windsurf/memories/`, scoped to the workspace they were created in. They are not committed to the repository and are not shared across machines.

### Retrieval mechanism

"Cascade retrieves them when it believes they're relevant." The exact retrieval algorithm is not documented publicly. Memories from one workspace are not available in another.

### What triggers storage

- **Auto-generation**: Cascade autonomously stores memories when it encounters context it deems worth remembering (toggleable via "Auto-Generate Memories" setting)
- **Explicit request**: User asks Cascade to "create a memory of..."
- **Rules**: Manually authored by developers

### Known limitations

Windsurf's AI "operates within session boundaries." The Cascade agent that spent time understanding project architecture, learning coding patterns, and mapping dependency structures "loses all of that context when you close the IDE or start a new session." The memories system partially mitigates this, but the documentation recommends preferring Rules over Memories for knowledge you want reliably reused, since auto-generated memories have less predictable retrieval.

## Cross-System Pattern Summary

| System | Storage format | Retrieval method | Who writes memory | Cross-session? | Auto-learning? |
|--------|---------------|-----------------|-------------------|----------------|----------------|
| **Reflexion** | Text strings in episodic buffer | Direct prompt injection (most recent) | LLM self-reflection model | Across trials (not sessions) | Yes (verbal reflection) |
| **Anthropic harness** | JSON (features) + text (progress) + git | Read file at session start | Agent writes at session end | Yes | No (structured protocol) |
| **Devin** | Knowledge items (trigger + content + macro) + notes.txt | Trigger-based contextual matching | Human + agent suggestions | Yes (Knowledge); partial (notes) | Yes (auto-suggestions) |
| **AutoGPT** | JSON file with embeddings | Numpy dot product similarity | Agent during think-execute cycle | Unreliable (file loss issues) | Yes (embedding-based) |
| **Cursor** | Markdown/MDC files with frontmatter | Static injection at session start | Human only | Yes (rules persist) | No |
| **Claude Code** | Markdown (CLAUDE.md + MEMORY.md + topic files) | Static injection + on-demand file reads | Human (CLAUDE.md) + agent (auto memory) | Yes | Yes (auto memory) |
| **Windsurf** | Opaque local files + markdown rules | "Relevance-based" (undocumented) | Human (rules) + agent (memories) | Yes (partial) | Yes (auto-generate) |
