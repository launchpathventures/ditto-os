# UX Interaction Spec: Mobile/Remote Agent OS Experience

**Date:** 2026-03-19
**Role:** Dev Designer
**Status:** reviewed — two review passes completed (main spec + persona validation), all feedback incorporated
**Feeds:** Dev Architect (runtime deployment decision, brief for mobile surface)
**Research inputs:** `docs/research/runtime-deployment-models.md`, `docs/research/input-type-taxonomies.md`, `docs/research/mobile-interfaces-for-agent-platforms.md`

---

## The Core Problem (User's Words)

> "I have no way of moving projects ahead without being at my computer."

This is not a feature request for a mobile app. It's a statement about **blocked decision flow**. Agents do work. That work needs human decisions — approve, reject, edit, clarify, prioritise. If the human can't make those decisions, the pipeline stalls. The agents are ready; the human is the bottleneck.

Every hour the process owner is away from their desk is an hour where:
- Completed work sits in a review queue, unreviewed
- Agents that need approval are paused, idle
- Context that could route work correctly lives only in the human's head
- Improvement proposals go stale
- The compound effect (human-layer.md: "it gets better every week") slows down

**The mobile experience is not about doing desktop work on a smaller screen. It's about keeping the decision pipeline flowing when you're not at your desk.**

---

## Insight: Mobile Is Operate Mode

The architecture defines two coexisting modes (architecture.md lines 23-31):
- **Explore** — discovery, refinement, debugging (conversation with a smart colleague)
- **Operate** — execution, monitoring, review, improvement (factory floor with dashboards)

Mobile maps cleanly to this split:

| Mode | Desktop | Mobile |
|------|---------|--------|
| **Explore** | Full experience — dual-pane conversation + process builder | Not needed — process creation is sustained, focused work |
| **Operate** | Full experience — dashboards, queues, deep analysis | **This is what mobile must serve** — decisions, status, capture |

You never set up a new process from your phone. You operate existing ones. The mobile experience is **Operate Mode, optimised for decisions and context**.

This means the mobile surface doesn't need all 16 primitives. It needs the primitives that serve the decision pipeline.

## The Six Human Jobs on Mobile

| Job | Desktop weight | Mobile weight | Why |
|-----|---------------|---------------|-----|
| **Orient** | Full | **High** | "What happened while I was away?" is the first mobile question |
| **Review** | Full | **Critical** | This is the bottleneck. Unblocking review from mobile unblocks the entire pipeline |
| **Define** | Full | **None** | Process creation requires sustained attention, dual-pane, complex editing |
| **Delegate** | Full | **Low** | Trust adjustments are infrequent, considered decisions. Exception: quick "auto-approve similar" |
| **Capture** | Available | **High** | Already designed for mobile (human-layer.md Primitive 12). Voice, photo, text |
| **Decide** | Full | **Medium** | Simple decisions (apply/dismiss improvement) work on mobile. Complex analysis is desktop |

**The mobile experience serves three primary jobs: Orient, Review, Capture — plus simple Decide.**

## Primitive Adaptation for Mobile

### Tier 1: Mobile-Critical (must work beautifully on phone)

**Primitive 1: Daily Brief**
- Human job: Orient
- Mobile adaptation: This IS the mobile home screen. Push notification at configured time ("Good morning. 3 items need review. 1 process degraded."). Opens to full brief. Tappable items link to review queue or process cards.
- Interaction states:
  - Empty: "All clear. No items need your attention." (This should feel good, not empty.)
  - Loading: Brief is generated by the PM agent — may take a moment. Show last brief with "Refreshing..." indicator.
  - Partial: Brief generated but some processes haven't reported yet. Show what's known, flag gaps.
- UX pattern source: Apple Weather app (glanceable, personalised, layered detail). Linear Mobile inbox (tap-to-act items).

**Primitive 5: Review Queue**
- Human job: Review
- Mobile adaptation: The primary mobile workspace. List of items awaiting human decision, sorted by urgency/blocking status. Each item shows: process name, output summary, pre-review status (harness verdict), confidence, urgency.
- Three interaction depths:
  1. **Notification-level** — Push notification with action buttons: [Approve] [Review]. For spot-checked items where the harness gave a clean bill of health. The user can approve without opening the app.
  2. **Queue-level** — Swipe gestures on list items. Swipe right = approve. Swipe left = defer/flag. Tap = open detail. For items that need a quick look but not deep review.
  3. **Detail-level** — Opens Output Viewer (Primitive 6). For items that need careful examination.
- Trust tier determines which depth is needed:
  - **Autonomous** — doesn't appear in mobile queue (exception only)
  - **Spot-checked** — notification-level or queue-level. Batch approve available.
  - **Supervised** — queue-level minimum. Must at least see the pre-review summary.
  - **Critical** — detail-level required. Cannot approve from notification.
- Interaction states:
  - Empty: "No items waiting for review. Agents are working." (Show what's in progress, not nothing.)
  - Loading: Skeleton cards with process name visible.
  - Error: "Couldn't load review queue. [Retry] [View cached]" — falls back to last-synced queue if available.
  - Queue overflow: When 20+ items, surface "Batch actions" — approve all passing, spot-check N.
  - Offline: Show cached queue with "Last synced: 5 min ago." Allow approve/reject, queue actions for sync.
- UX pattern sources: Linear Mobile (swipe-to-triage inbox). Gmail (swipe actions, batch operations). iOS notification action buttons (approve without opening app). GitHub Mobile (PR review: summary-first, detail-on-demand).

**Primitive 6: Output Viewer (mobile variant)**
- Human job: Review
- Mobile adaptation: Simplified rendering optimised for decision-making, not editing. Shows: pre-review summary (what the harness already checked), output summary (not full output), flagged issues, and decision buttons.
- Output type adaptations for mobile:
  - **Text** — Shows pre-review annotations + excerpt. "See full output" expands. Inline editing NOT available on mobile — if the user needs to edit, they tap "Edit at desk" which flags it for desktop review and moves to the next item.
  - **Data** — Shows flagged rows only (exceptions). Clean rows summarised as "42 of 45 rows matched, 3 flagged." Tap flagged row for agent's reasoning.
  - **Code** — Shows harness review annotations (convention check, bug hunter, security). Diff summary ("3 files, 142 lines added"). Actual diff available but optimised for vertical scroll. Line-level comments NOT available on mobile.
  - **Action** — Shows what was done + result. Undo available if reversible.
  - **Decision** — Shows reasoning summary + conclusion. Override available.
- Key design rule: **On mobile, the human reviews the review, not the raw output.** The harness's pre-review is the primary content. The raw output is available but secondary.
- Interaction states:
  - Loading: Show process name + pre-review status while output loads.
  - Error: "This output couldn't be rendered on mobile. [Open at desk]"
  - Offline: Cached output available. Approve/reject queued for sync.
- UX pattern sources: GitHub Mobile (code diff: file-by-file navigation, summary-first). Superhuman email (4D triage: Do, Delegate, Defer, Delete).

**Primitive 7: Feedback Widget (mobile variant)**
- Human job: Review
- Mobile adaptation: Implicit feedback still captured — approve = positive signal, reject = negative signal. But the "Teach this" pattern detection and the diff-based feedback from editing are desktop features. On mobile, the feedback widget surfaces as:
  - Post-approval: "You've approved 5 similar outputs this week. [Auto-approve similar?]" — this is trust building from mobile.
  - Post-rejection: Quick reason picker (not freeform text): "Wrong tone" / "Missing info" / "Factually incorrect" / "Other" — tappable chips, not a text field. This preserves implicit feedback capture in a mobile-friendly format. **These reason chips must produce structured signals that the Learning Layer (L5) can act on** — they are not just labels. Each chip maps to a correction category that feeds pattern detection and quality criteria refinement, the same way desktop edit diffs do.
- Interaction states:
  - Pattern detected: Card appears after approval with the "Auto-approve similar" prompt. Dismissible.
  - No pattern: Nothing shown — feedback was captured silently via the approve/reject action.
- UX pattern sources: Hey.com (screen sender once, applies to all future messages — same as "auto-approve similar"). Apple Photos (post-action suggestions that learn from behaviour).

**Primitive 12: Quick Capture**
- Human job: Capture
- Mobile adaptation: Already designed for mobile in human-layer.md. The primary mobile input surface.
- Refinements for this spec:
  - Voice-first on mobile. Large microphone button. Tap to record, tap to stop. Transcription shown immediately.
  - Photo capture with context: take a photo, the system extracts text/content and routes.
  - **The system classifies, the user confirms.** Research finding: no AI agent platform requires users to pre-classify inputs. The GTD insight: first question is binary (actionable vs not). The system should auto-classify and show: "This looks like a task for [Delta project]. [Correct] [Change]"
  - Multi-modal: text, voice memo, photo, link, file attachment, screenshot share.
  - Always accessible: floating action button (FAB) on every mobile view.
- Interaction states:
  - Recording: Waveform animation. Duration counter.
  - Transcribing: "Transcribing..." with shimmer animation on the text area.
  - Classifying: "Routing to [Project]..." — brief, dismissible. User can override.
  - Success: "Captured. Will appear in tomorrow's brief." or "Added to [Process] context." Dismissible toast.
  - Transcription failed: "Couldn't transcribe. Saved as voice memo. [Retry] [Keep as voice memo]" — the capture is never lost, even if AI processing fails.
  - Classification failed: "Couldn't determine where this belongs. [Pick a project] [Just save it]" — falls back to inbox-style unrouted capture.
  - Offline: Captured locally, "Will sync when connected" badge. Voice recorded locally, transcription queued.
- UX pattern sources: Apple Reminders via Siri (voice → task with zero friction). Things 3 inbox (capture first, organise later). Hey.com set-aside (park it, deal later).

**Primitive 4: Performance Sparkline**
- Human job: Orient
- Mobile adaptation: Unchanged — these are already tiny by design. They appear inside Process Cards and Agent Cards. On mobile, they provide the glanceable "is this getting better or worse?" signal.

### Tier 2: Mobile-Useful (simplified versions)

**Primitive 2: Process Card (glance only)**
- Human job: Orient
- Mobile adaptation: Glance view only (no expanded view on mobile). Shows: name, status, health, trust tier, last run. Tap → opens process detail on desktop (or a simplified status screen on mobile showing: recent activity + sparklines + next scheduled run).
- Appears in: Daily Brief, Process Health section of home view.

**Primitive 3: Activity Feed (filtered)**
- Human job: Orient
- Mobile adaptation: Shows recent activity, **system-filtered to what's actionable or notable** — not the full firehose. The system selects what to show (completions, flags, failures, human actions). This differs from desktop where the user controls filters. On mobile, the user can filter by process (tap a process name) but not by event type or time range — that granularity is desktop.
- Appears in: Process status screen (when you tap a Process Card). Pull-to-refresh.

**Primitive 13: Improvement Card**
- Human job: Decide
- Mobile adaptation: Shows diagnosis + evidence + suggestion. Three actions: [Apply] [Dismiss] [Discuss later]. Note: human-layer.md defines four desktop actions: Apply, Modify, Dismiss, Discuss. Mobile intentionally omits **Modify** (requires complex editing, desktop-only) and renames **Discuss** to **Discuss later** (defers to Conversation Thread on desktop, since Conversation Thread is a desktop-only primitive). Cannot modify the suggestion on mobile — only accept, reject, or defer.
- Appears in: Review Queue (improvement proposals are review items).

### Tier 3: Desktop-Only

**Primitive 8: Conversation Thread** — Requires sustained attention, typing, dual-pane layout. Desktop only.
**Primitive 9: Process Builder** — Complex structured editor. Desktop only.
**Primitive 10: Agent Card** — Infrequent access, detailed performance analysis. Desktop only (appears as simplified reference in Activity Feed).
**Primitive 11: Trust Control** — Considered decision with evidence analysis. Desktop only. Exception: the "Auto-approve similar" prompt from the Feedback Widget is mobile-native trust building.
**Primitive 14: Process Graph** — Visual dependency map. Desktop only (too complex for mobile, too infrequent).
**Primitive 15: Data View** — Charts and tables. Desktop only (data analysis is sustained work).
**Primitive 16: Evidence Trail** — Source verification. Desktop only (drill-down research).

## Push Notification Design

Notifications are the mobile experience's most important interaction surface. The user may never open the app if notifications are well-designed.

### Notification Types

| Type | When | Actions | Opens to |
|------|------|---------|----------|
| **Morning brief** | Configured time (e.g., 7:30am) | [Open brief] | Home view |
| **Review ready** | Output enters queue at supervised/spot-checked tier | [Approve] [Review] | Review detail |
| **Batch ready** | 3+ routine items in queue | [Approve all] [Spot-check] | Review queue |
| **Process alert** | Health degrades or process fails | [View] | Process status |
| **Improvement proposal** | Learning layer surfaces suggestion | [Apply] [Dismiss] | Improvement card |
| **Capture reminder** | User deferred something to "later today" | [Capture now] | Capture screen |

### Notification Priority Levels

| Priority | Notification behaviour | Example |
|----------|----------------------|---------|
| **Urgent** | Sound + badge + banner | Critical-tier output needs review; process failed |
| **Standard** | Badge + banner (silent) | Supervised output ready for review |
| **Batch** | Badge only (grouped) | Multiple spot-checked items accumulated |
| **Ambient** | Badge count update only | Morning brief ready; improvement proposal |

### Key Design Rule: Notifications Should Reduce App Opens, Not Increase Them

A well-designed notification lets the user make a decision without opening the app. "Weekly report ready. Confidence: 94%. All checks passed. [Approve] [Review]" — if the user trusts the harness's assessment, they tap Approve and continue their day. The pipeline advances. The app was never opened.

This is the "auto-approve similar" trust building in miniature: every notification-level approval is an implicit signal that the user trusts this process at this quality level.

## The "Edit @ Desk" Pattern (Original to Agent OS)

A key mobile interaction that no existing tool implements well:

**Situation:** The user is reviewing an output on mobile. The pre-review flagged an issue. The user agrees it's an issue but can't fix it on a phone (e.g., editing code, restructuring a document, adjusting data).

**Pattern:**
1. User taps "Edit @ desk"
2. The item is flagged in the queue as "Acknowledged — pending desktop edit"
3. The pipeline is NOT blocked for other items — only this specific output is held
4. When the user opens the desktop dashboard, this item is surfaced prominently: "You flagged this for editing from mobile"
5. The user makes the edit on desktop
6. Feedback is captured as normal (the edit IS the feedback)

**Post-action state on mobile:**
- The item moves from the main queue to a "Flagged for desk" section at the bottom of the Review Queue (visually distinct, muted, with a 🖥 icon)
- The item shows: "Acknowledged by you — pending desktop edit" with timestamp
- The item does NOT count toward the "items need review" badge — the decision has been made, execution is deferred
- If a downstream process depends on this output, the dependency graph (L4) should know this item is in "human hold" state — not "pending review" (no one is looking at it) but "pending human edit" (the human has committed to acting). This is a distinct state that prevents the harness from escalating or re-queuing.

**Why this matters:** Without "Edit @ desk", the user's only mobile options are approve (which they can't do in good conscience) or reject (which sends it back to the agent unnecessarily). "Edit @ desk" acknowledges the issue, keeps the pipeline flowing for everything else, and creates a clean handoff to desktop.

**This is mobile-to-desktop continuity as a first-class interaction, not an afterthought.**

## The Mobile Day: A Process Owner's Experience

### 7:30am — Push notification

> "Good morning. 3 items ready for review. Auth flow for Delta has 1 flagged issue."

The process owner is making coffee. They glance at the notification. They know about the auth flow — they briefed it yesterday. One flagged issue? That could wait, or it could be quick.

### 7:35am — Notification triage

They long-press the notification. The expanded view shows:
- Auth flow: ⚠ 1 convention issue (raw `<button>`)
- Weekly report: ✓ all checks passed
- Lead scoring batch: ✓ 23 leads, all within parameters

They tap [Approve] on the weekly report directly from the notification. They tap [Approve all] on the lead scoring batch. Two items unblocked in 10 seconds without opening the app.

### 7:40am — Open app for the flagged item

They tap [Review] on the auth flow notification. The Review Detail screen shows the pre-review summary. Convention checker flagged a raw `<button>`. They agree — it should be the `Button` component. But they can't edit code from their phone.

They tap [Edit @ desk]. The item is acknowledged. They scroll down — nothing else needs attention.

**Three outputs reviewed. Two approved. One flagged for desk. Total time: 2 minutes. Pipeline flowing.**

### 10:00am — Walking between meetings

A thought: "The insurance platform should use the same auth pattern as Delta."

They swipe down to the Capture FAB and tap the microphone. "The insurance platform should use the same auth pattern as Delta. Check if the CRM integration needs it too."

The system transcribes, classifies (cross-project dependency note, routes to both Delta and Insurance projects), and confirms with a toast: "Captured. Routed to Delta + Insurance."

### 2:00pm — Notification

> "Improvement proposal: Lead scoring accuracy dropped 15% since LinkedIn source added. [Apply fix] [Dismiss]"

They tap [Apply fix] from the notification. The learning layer's proposed criteria are applied. No app open needed.

### 6:00pm — At the desk

They open the desktop dashboard. The auth flow item is surfaced at the top: "You flagged this for editing from mobile at 7:40am." They open it, fix the `<button>` → `Button` issue inline, approve. The feedback widget captures the edit.

**The day's outcome: every output was reviewed, every decision was made, context was captured, an improvement was applied. The process owner was at their desk for 15 minutes of agent-related work. The rest happened from their phone.**

## Input Classification: How Capture Becomes Action

The research found six classification dimensions (input-type-taxonomies.md) and a universal finding: **no AI agent platform requires users to pre-classify inputs. Classification is the system's job.**

### The Agent OS Classification Model (Process Owner's Perspective)

When the user captures something, the system runs the Capture → Classify → Route pipeline (already defined in human-layer.md, Primitive 12). From the user's perspective:

**Step 1: Capture** — User dumps text, voice, photo, link. Zero structure required.

**Step 2: System classifies** — The system determines:
- **Which process/project?** (Context-aware: what were they looking at? What's active?)
- **Actionable or context?** (The GTD binary: does this need action, or is it information?)
- **If actionable, what kind?** (New task, bug report, feature request, goal, decision needed)
- **Urgency?** (Now, scheduled, someday)

**Step 3: System confirms** — Brief confirmation shown:
- "Task for Delta project: 'Fix auth pattern for CRM'. [Correct] [Change]"
- Or: "Context for Insurance project. Will surface in next brief. [Correct] [Change]"

**Step 4: User corrects or confirms** — One tap. If the system got it wrong, the correction IS feedback that improves future classification.

### What the User Sees (Not What the System Does)

The user never sees "input type taxonomy" or "classification dimensions." They see:

- "Sounds like a task for [Project]. Right?" → [Yes] [No, it's just context] [No, different project]
- "This seems urgent — should it go to today's brief?" → [Yes] [No, whenever]
- "I see this mentions [Process]. Should I add it there?" → [Yes] [No, it's separate]

**The system proposes, the user confirms.** One question at a time (boiling frog principle from human-layer.md). Most captures need zero corrections — the system's classification improves over time from the corrections it does receive.

### Standard Input Types (What the System Uses Internally)

Based on the research, these are the input types the system classifies into. The user never selects these — the system infers them:

| Input type | What it means (user mental model) | Process routing |
|-----------|----------------------------------|----------------|
| **Task** | "Something that needs to be done" | → Creates work item in relevant process |
| **Bug** | "Something is broken" | → Bug Investigation process (Process 4) |
| **Context** | "Here's information the system should know" | → Attaches to project/process context |
| **Goal** | "Here's what I want to achieve" | → Surfaces in Project Orchestration (Process 5) for planning |
| **Decision** | "I need to decide something" | → Surfaces in Daily Brief as decision item |
| **Feedback** | "This could be better" | → Routes to relevant process's improvement pipeline |

These map to the GTD model: Task/Bug = "next action or project", Context = "reference", Goal = "project outcome", Decision = "waiting for (self)", Feedback = "someday/maybe improvement."

They also map to ITIL: Task = "service request", Bug = "incident", Feedback = "problem/change request."

**The key principle: the taxonomy exists for routing, not for the user.** The user speaks naturally; the system classifies.

## Interaction States: The Offline and Low-Connectivity Experience

Mobile means unreliable connectivity. The experience must gracefully handle offline and partial connectivity.

### Offline Capability Requirements

| Action | Offline behaviour | Sync behaviour |
|--------|------------------|----------------|
| **View Daily Brief** | Show last-synced brief with "Last updated: [time]" | Refresh on reconnect |
| **View Review Queue** | Show cached queue items | Refresh on reconnect, reconcile approvals |
| **Approve/Reject** | Queue action locally, show "pending sync" badge | Sync on reconnect; if item was modified server-side, show conflict |
| **Capture** | Save locally (text/voice/photo) | Sync on reconnect; classify server-side |
| **View Process Health** | Show last-synced status | Refresh on reconnect |

### Conflict Resolution

What happens if you approve something from mobile while offline, but the agent re-ran and produced a new version before you came back online?

**Design principle: human intent wins, but the system must inform.**

1. Your offline approval is queued
2. On reconnect, the system detects the output has changed
3. It shows: "The auth flow output was updated while you were offline. Your approval was for the previous version. [Review new version] [Approve anyway]"
4. The user decides

This preserves human agency without silently applying stale decisions.

## Process Architecture: Capture → Classify → Route

This is already described in human-layer.md as a five-step pipeline. The mobile experience makes this process more prominent — it's one of the most-used mobile interactions.

### How the Process Owner Thinks About It

The user doesn't think "I'm feeding a process pipeline." They think: "I had a thought. I dumped it into Agent OS. It'll show up where it needs to."

The Capture → Classify → Route process should be:
- **Invisible when it works** — capture, confirmation toast, done
- **Gently correctable when it doesn't** — "I routed this to Delta, but it should be Insurance. [Change]"
- **Self-improving** — corrections feed the classification model. After 50 captures, it should get it right 90%+ of the time

### Process Definition (How It Would Look in YAML)

Not a technical design — but from the user's perspective, this is a pre-configured process that every user has:

```
Process: Capture → Classify → Route
├── Trigger: Human capture (text, voice, photo, link)
├── Steps:
│   1. Transcribe (if voice)               [Script]
│   2. Classify: project, type, urgency    [AI Agent]
│   3. Confirm with user                   [Human — lightweight]
│   4. Route to destination                [Script]
│   5. Surface in next Daily Brief         [Script]
├── Trust: Autonomous (corrections are implicit feedback)
└── Quality: Classification accuracy tracked via correction rate
```

The user never sees this process definition. They just use Quick Capture. But the harness runs, trust accumulates, and the classification improves.

## What This Spec Does NOT Address (Architect's Domain)

This spec intentionally avoids:
- **Where the runtime lives** (cloud, local, hybrid) — the Architect decides based on UX requirements + technical constraints
- **PWA vs native** — the Architect evaluates feasibility of the interaction patterns described here
- **Push notification infrastructure** — the Architect designs the delivery mechanism
- **Sync protocol** — the Architect chooses how offline actions reconcile
- **Authentication mechanism** — the Architect selects passkeys/OAuth/session tokens based on security requirements

**What this spec gives the Architect:**
1. Mobile is Operate Mode — the runtime must be accessible when the user is away from their desk
2. Push notifications with action buttons are the primary mobile surface — the runtime must support push delivery
3. Offline approve/reject must work — the data model must support queued actions
4. "Edit @ desk" creates a mobile → desktop handoff — the system must track item state across devices
5. Capture must work offline — local storage + sync is required
6. The Capture → Classify → Route pipeline needs an always-available AI classifier — it can't depend on the user's laptop being open

**These are UX requirements that constrain the runtime decision.** The Architect should read them as: "Whatever runtime model you choose, it must support these six capabilities."

## Design Discoveries

Two insights emerged during this work that should be captured:

### Emerging Insight: Mobile Is Operate Mode
Desktop serves both Explore and Operate. Mobile serves Operate only, with Capture as the bridge. This is not a limitation — it's a natural fit. You don't design processes on your phone. You make decisions, provide context, and keep the pipeline flowing.

### Emerging Insight: "Edit @ Desk" Is a First-Class Interaction
No existing tool handles the mobile-to-desktop handoff well for complex review items. Agent OS's "Edit @ desk" pattern — acknowledge, flag, don't block, resume at desk — is original and solves a real gap. This is not "save for later" (passive) — it's "I've seen this, I agree with the issue, I'll fix it when I have the right tools" (active acknowledgment).

## Persona Validation

Testing the mobile experience against each of the four personas defined in `docs/personas.md`. For each persona: does the mobile spec serve their specific reality? Would they actually use it the way described? What's missing?

### Rob — Trades MD (on job sites 60% of the day)

**Rob's mobile context:** Phone in one hand, tools in the other. Driving between jobs. Lunch break in the truck. Kitchen table at 6:30pm. Rob is the most mobile-dependent persona — his phone IS his office during work hours.

**What the spec gets right for Rob:**
- **Morning brief in the truck (7:30am push notification):** Maps directly to Rob's 6:30am phone check. He currently scans 23 emails and 8 texts. The Daily Brief replaces this chaos with prioritised, reasoned recommendations.
- **Notification-level approve:** Rob approves quotes between jobs. "New quote ready for Acme bathroom reno — $14,200. [Approve] [Review]" — this is the 30-second decision that currently waits until 6:30pm at the kitchen table.
- **Voice capture on site:** Rob currently scribbles on receipts. Voice capture ("Henderson also wants HW quote, Rinnai system, access is tight") replaces lost context. The system classifies and routes — Rob doesn't think about it again.
- **"Edit @ desk":** Rob's "desk" is the kitchen table. He flags a quote that needs a labour adjustment on his phone at lunch, fixes it at 6:30pm. The pipeline doesn't wait 8 hours for the adjustment — other quotes keep flowing.

**What's missing for Rob:**

1. **Hands-busy context (on-site, carrying materials, gloved hands).** Rob is frequently in situations where he can't easily use a touchscreen — on a ladder, holding a pipe, wearing gloves. The spec has voice capture but doesn't address voice-driven CONSUMPTION. Could the morning brief be read aloud? Could Rob say "approve the Acme quote" while his hands are full? The mobile research found that Siri App Intents (native only) and Android App Actions can register voice commands. This is a UX requirement the spec should flag: **voice-in AND voice-out for the hands-busy process owner.** This doesn't change the spec's design — it adds a seventh UX constraint for the Architect: the runtime/platform choice must not preclude voice interaction (PWA cannot do this; native can).

2. **Customer-triggered process visibility.** Rob's processes are triggered by customer inputs — a call, an email, a text asking for a quote. The spec's mobile home view shows items in the Review Queue but doesn't explicitly show "what triggered this." Rob needs to see: "Acme bathroom reno — triggered by email from Sarah Jones, 2:30pm." This is context that helps Rob make better mobile decisions. The Output Viewer's pre-review summary should include trigger context, not just harness review results.

3. **Response urgency signalling.** Rob says: "The guy who calls back first gets the job." The spec has urgency indicators ("Items that block other processes or have deadlines surface higher") but doesn't address time-to-respond urgency for customer-facing processes. Rob needs: "Quote requested 4 hours ago. Average competitor response: 6 hours." This is a process-specific urgency signal that the Daily Brief and Review Queue should surface.

**Verdict:** The spec serves Rob well for his core needs (approve, capture, orient). The three gaps are additive, not structural — they enhance Rob's experience without changing the spec's architecture.

### Lisa — Ecommerce MD (warehouse, meetings, home office)

**Lisa's mobile context:** Moving between warehouse, supplier meetings, and home office. Phone for status checks between contexts. Home office desk for deep work. Lisa is mobile 40% of the time, desk 60%.

**What the spec gets right for Lisa:**
- **Morning brief (7:00am phone check):** Replaces Lisa's Shopify dashboard scan. "2 descriptions ready for review. 3 products flagged for pricing — competitor X dropped widget price 20%." More actionable than raw dashboard metrics.
- **Photo capture at warehouse:** Lisa screenshots a negative review and texts customer service. With Agent OS, she photos the review, the system classifies it ("customer complaint — wrong size shipped"), routes it to the customer service process, and it surfaces in the right review queue. No more texting customer service.
- **"Edit @ desk" for content:** Lisa's corrections to product descriptions are taste-driven rewrites, not simple approve/reject. "Edit @ desk" is perfect — she flags "this description is too generic" from her phone and rewrites it at her home office.

**What's missing for Lisa:**

1. **Taste-based feedback on mobile.** Lisa's corrections are nuanced — she doesn't just reject, she rewrites to "highlight the differentiators" and "match our voice." The spec's mobile rejection chips ("Wrong tone" / "Missing info" / "Factually incorrect" / "Other") are too blunt for Lisa's feedback. Lisa needs a way to express *what's wrong* with more nuance on mobile — even if she can't fix it there. Option: a brief voice note attached to "Edit @ desk": "The description is too generic — needs to emphasise the handmade aspect and the Italian leather source." This voice note becomes context for when she edits at her desk, AND it feeds the Learning Layer as a richer signal than a chip. **The spec should add voice annotation as a mobile feedback mechanism alongside the reason chips.**

2. **Competitive context in review.** Lisa reviews pricing and content with competitor awareness. The mobile Output Viewer should support contextual data alongside the output being reviewed — "your price: $89. Competitor A: $79 (dropped 12% this week). Competitor B: $95." This is process-specific context that the harness should assemble and the mobile viewer should render as a compact data card alongside the output.

**Verdict:** The spec serves Lisa well for orient and capture. The taste-feedback gap is the most significant — Lisa's corrections are the primary source of learning data for her processes, and the mobile feedback mechanism needs more nuance than chips alone. Voice annotation solves this elegantly.

### Jordan — Generalist Technologist (desk-based primary)

**Jordan's mobile context:** Commute, between meetings, occasional status checks. Jordan is the least mobile-dependent persona — the personas doc explicitly says "Mobile is secondary for Jordan."

**What the spec gets right for Jordan:**
- **Morning brief on the commute:** Jordan checks process health across departments — "Finance reconciliation: green. HR references: green. Marketing reports: amber (2 failures this week)." Quick commute scan.
- **Mobile is Operate Mode only:** Jordan's primary activities (Define processes, demo Process Graph to leadership, cross-department analysis) are all desktop. The spec's explicit exclusion of Explore mode from mobile is validated by this persona.
- **Capture between meetings:** Jordan overhears a department head mention an automation opportunity. Quick text capture routes it to Jordan's backlog.

**What's missing for Jordan:**

1. **Multi-process / cross-department dashboard.** Jordan runs 4+ processes across 3+ departments. The mobile home view's "Process Health" section shows a flat list of processes. Jordan needs a grouping — by department, by status, by urgency. The spec should note that Process Health on mobile should support lightweight grouping (e.g., collapsible department headers) for users managing many processes. This doesn't require a new primitive — it's a composition option for the Process Card glance view.

**Verdict:** The spec serves Jordan well. Jordan validates the "mobile is Operate mode" design. The single gap (process grouping) is minor and cosmetic.

### Nadia — Team Manager (governs quality across team)

**Nadia's mobile context:** Before standup, between 1:1s, before client meetings. Phone for team status checks. Desk for quality review and team development.

**What the spec gets right for Nadia:**
- **Pre-standup morning brief:** "Team output: 6 reports delivered, 4 approved clean, 2 corrected. Chen's data gathering process: 3 failures this week." Nadia walks into standup already knowing what happened — no more Slacking each team member for updates.
- **Review Queue for team governance:** Nadia reviews outputs from multiple team members' processes. The queue surfaces what needs her judgment.
- **Trust delegation from mobile:** System suggests upgrading an analyst's formatting process to spot-checked. Nadia can accept/decline from her phone via push notification.

**What's missing for Nadia:**

1. **Team-level attribution in the mobile queue AND trust notifications.** Nadia isn't reviewing her own process outputs — she's governing across her team. The Review Queue items need to show WHOSE process produced this: "Chen — quarterly report formatting" not just "quarterly report formatting." The spec's wireframe shows process name but not team-member attribution. For Nadia, the person matters as much as the process — she tracks quality per analyst, not just per process. **This attribution also applies to trust delegation notifications** — when the system suggests upgrading trust, Nadia needs to see "Chen's quarterly report formatting is ready for spot-checked" not just "quarterly report formatting is ready for spot-checked." Trust is per-process, per-team-member (personas.md lines 193-194).

2. **Team health view on mobile.** The spec's Process Health section shows individual process status. Nadia needs a team-level roll-up: "Chen: 3 processes, all green. Priya: 2 processes, 1 amber. Alex: 2 processes, all green." This is a glanceable team dashboard that sits between the Daily Brief and the individual Process Cards. It maps to the "Team" view composition from human-layer.md, but the spec doesn't adapt this for mobile. **A simplified Team Health primitive (roll-up by person) should be a Tier 2 mobile-useful item.**

3. **Coaching signals.** Nadia's job isn't just approval — it's developing her team. When she sees "Chen's correction rate is 40% on data sourcing" on mobile, she needs to know: is this getting better or worse? The Performance Sparkline serves this, but it should be visible in the team health context, not just per-process. Nadia wants to see at a glance: "Chen's accuracy is trending up across all processes" or "Priya's formatting corrections are increasing — might need a coaching conversation."

**Verdict:** The spec under-serves Nadia compared to the other three personas. The core issue is that the spec was designed for a single process owner, but Nadia is a multi-process, multi-person governor. The team attribution and team health gaps are significant for her mobile experience — without them, she falls back to Slacking team members for status, which is exactly what Agent OS should eliminate.

## Summary: Persona Validation Results

| Persona | Core needs met? | Gaps found | Severity |
|---------|----------------|------------|----------|
| **Rob** (trades, on-site) | Yes — approve, capture, orient all work | Voice-out (brief read aloud), trigger context, response urgency | Medium — additive, not structural |
| **Lisa** (ecommerce, mixed) | Yes — orient, capture, edit @ desk | Taste feedback via voice annotation, competitive context in review | **Medium-High** — the voice annotation directly affects L5 learning signal quality; without it, Lisa's taste corrections (her primary feedback type) produce degraded signals on mobile |
| **Jordan** (technologist, desk) | Yes — validates mobile = Operate | Process grouping for multi-department view | Low — cosmetic |
| **Nadia** (team manager) | Partially — individual process ops work, team governance doesn't | Team attribution in queue, team health roll-up, coaching signals | **High — spec needs revision for this persona** |

### Design Recommendations

1. **Add voice annotation to "Edit @ desk" and reject flows.** When the user flags an item for desktop editing or rejects, they can optionally record a brief voice note explaining why. This serves Lisa's taste-based feedback AND feeds the Learning Layer with richer signals than chips alone. Voice annotation is already natural on mobile — it's a short extension of the existing voice capture capability.

2. **Add team-member attribution to Review Queue items.** Each item shows: "[Person] — [Process name]" for multi-person governors like Nadia. This is a data display change, not a new primitive.

3. **Add a simplified Team Health section to the mobile home view (Tier 2).** Roll-up by person: name, number of processes, aggregate health status, performance sparkline. This serves Nadia's pre-standup scan and maps to the existing Team view composition from human-layer.md (which uses Agent Cards + Performance Sparklines). The Architect should determine whether this reuses the Agent Card primitive (which can represent humans per architecture.md line 437) in a mobile composition, or requires a new person-level roll-up component.

4. **Flag voice-driven interaction as a seventh Architect constraint.** The platform/runtime choice must not preclude voice commands (approve/reject) and voice consumption (brief read aloud). This is critical for Rob and beneficial for all personas in hands-busy contexts. PWA cannot do this; native can.

5. **Add trigger context to the Output Viewer's pre-review summary.** Show what triggered the process run alongside the harness review results. This helps Rob ("customer email from Sarah Jones, 2:30pm") and Lisa ("new product added to catalogue, supplier: Artisan Leather Co.") make better mobile decisions.

6. **Add response-urgency signals for customer-facing processes.** Process definitions can declare a target response time. The Review Queue surfaces items approaching their target with urgency markers. This is process metadata, not a new primitive.
